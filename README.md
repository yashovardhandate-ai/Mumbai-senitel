# Mumbai Sentinel — deployment guide

A live incident-reporting map for Mumbai: report traffic jams, blockages,
fires, floods, or other problems, with community upvote/downvote to surface
real reports. Built with Vite + React + Leaflet (OpenStreetMap) + Supabase.

## 1. Create a Supabase project (free)

1. Go to https://supabase.com and sign up / log in.
2. Click **New project**, pick a name and a database password, and wait for
   it to finish provisioning.

## 2. Create the tables

Open the **SQL Editor** in your Supabase project and run this:

```sql
create table incidents (
  id text primary key,
  category text not null,
  description text not null,
  reporter_name text not null,
  reporter_phone text not null,
  lat double precision not null,
  lng double precision not null,
  resolved boolean not null default false,
  owner_token text not null,
  created_at timestamp with time zone default now()
);

create table votes (
  id uuid primary key default gen_random_uuid(),
  incident_id text not null references incidents(id) on delete cascade,
  voter_name text not null,
  direction text not null check (direction in ('up', 'down')),
  created_at timestamp with time zone default now(),
  unique (incident_id, voter_name)
);

alter table incidents enable row level security;
alter table votes enable row level security;

create policy "Public read incidents" on incidents for select using (true);
create policy "Public insert incidents" on incidents for insert with check (true);
create policy "Public update incidents" on incidents for update using (true);

create policy "Public read votes" on votes for select using (true);
create policy "Public insert votes" on votes for insert with check (true);
create policy "Public update votes" on votes for update using (true);
```

This gives every incident a unique ID (generated by the app itself using a
timestamp) and every vote a row tied to a specific incident and a specific
name — the `unique (incident_id, voter_name)` line is what makes one person
unable to vote twice on the same report; voting again just changes their
existing vote instead of adding a new one.

## 3. Get your API keys

In Supabase: **Settings → API** → copy the **Project URL** and the
**anon public** key.

If working locally: copy `.env.example` to `.env` and paste them in. If
deploying straight to Vercel (recommended on iPad), skip this and enter them
directly in Vercel's environment variables instead (step 6).

## 4. Push to GitHub

Create a new repo on github.com, then either:
- **On a computer:** `git init`, `git add .`, `git commit -m "Initial commit"`,
  `git remote add origin <your-repo-url>`, `git push -u origin main`
- **On iPad:** use **Add file → Create new file** for each file, typing the
  full path (e.g. `src/App.jsx`, `src/lib/supabaseClient.js`) so GitHub
  creates the folders automatically. Skip `.env` and `.gitignore`.

## 5. Deploy on Vercel (free)

1. vercel.com → sign in with GitHub → **Add New → Project** → import this repo
2. Before deploying, add environment variables:
   - `VITE_SUPABASE_URL`
   - `VITE_SUPABASE_ANON_KEY`
3. Click **Deploy**

You'll get a live URL in a minute or two.

## What's included in this version

- Live Mumbai map (OpenStreetMap tiles via Leaflet)
- Report an incident: drop a pin, pick a category, describe it, give name + phone
- Emoji markers color-coded by category, with a category filter bar
- Sidebar feed sorted by community score (upvotes minus downvotes)
- Named voting — one vote per person per incident, remembered in the browser
  so they don't have to type their name every time
- A "new incidents" badge that polls every 15 seconds
- **Reports auto-hide after 24 hours** even if nobody resolves them
- **"Mark resolved" button** — visible only to the person who submitted the
  report (their browser holds a private ownership token nobody else has).
  This is enforced the same way as everything else in this app: it's a
  reasonable deterrent, not bulletproof server-side security. Someone with
  real technical intent could still bypass it. If this app gets real public
  usage, the next step would be moving that check into a server-side
  function instead of relying on a client-held token.

## What's NOT included yet (future work)

- **Directory of police/fire/BMC/PWD contact numbers** — planned for a later
  version
- **Real push notifications** — right now "alerts" just means the badge count
  updates while the site is open; true push notifications to a phone would
  need a separate service (e.g. web push or an SMS/WhatsApp integration)
- **Moderation beyond voting** — there's no way to delete a false report
  except directly in the Supabase table editor. If bad reports become a
  problem, ask Claude to add an admin view or auto-hide incidents that drop
  below a certain score.
- **Phone number verification** — anyone can type any name/number; nothing
  currently confirms it's real
